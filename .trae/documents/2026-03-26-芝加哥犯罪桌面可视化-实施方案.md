# 芝加哥犯罪数据桌面可视化项目实施方案（规划稿）

## 1. 目标与边界

### 1.1 项目目标
- 基于课程任务，完成芝加哥犯罪数据集的数据库导入、分析与可视化展示。
- 采用桌面端形态交付，技术栈为 Electron + React + TypeScript + Vite + Ant Design + D3.js + Python 后端。
- 至少完成 10 个“分析角度不同”的统计分析，并提供交互式可视化。
- 输出可答辩的完整交付包：可运行程序、实验报告、汇报 PPT、演示脚本与源码。

### 1.2 本次规划边界
- 当前阶段只做详细方案与任务拆解，不改代码、不初始化工程、不执行实现。
- 以“可直接进入开发”为目标，明确模块、接口、目录、里程碑、验收标准与风险控制。

## 2. 需求抽取（来自任务书与现有调研）

### 2.1 课程硬性要求
- 数据集：Chicago Crime Dataset。
- 数据落库：导入关系型数据库（MySQL）。
- SQL 分析：不少于 10 条，且分析角度需要本质不同。
- 展示方式：柱状图、折线图、饼图等；前后端交互可视化可加分。
- 成果提交：报告、PPT、源码，且需线下汇报。

### 2.2 高分导向要求
- 不只“出图”，每个分析都要给出“结论 + 解释 + 对比”。
- 前后端交互要体现“筛选、联动、反馈”，而不是静态图拼接。
- 工程完整性要可证明：可运行、可复现、可打包、可演示。

## 3. 技术选型与架构决策

### 3.1 选型结论
- 桌面容器：Electron（跨平台、演示稳定）。
- 前端：React + TypeScript + Vite（开发效率高、类型安全、热更新快）。
- 组件库：Ant Design（快速搭建业务型数据看板）。
- 可视化：D3.js（细粒度交互能力强，适合课程展示“可视化实现能力”）。
- 后端：Python FastAPI（接口开发快、文档友好、与数据分析生态兼容）。
- 数据访问：SQLAlchemy + PyMySQL（连接 MySQL，结构清晰）。

### 3.2 架构原则
- 分层解耦：Main/Preload/Renderer 与 Backend 分离，前后端通过 HTTP API 通信。
- 安全默认：Renderer 不直接访问 Node 能力，Preload 做白名单桥接。
- 契约优先：先定 API 输入输出，再做图表组件与后端查询实现。
- 数据前置聚合：后端返回“可直接渲染”的统计结果，降低前端二次计算复杂度。

### 3.3 建议目录结构
- `frontend/`：Electron + React 前端工程
- `frontend/electron/`：主进程与 preload
- `frontend/src/`：页面、组件、图表、服务层
- `backend/`：FastAPI 服务
- `backend/routers/`：分析接口路由
- `backend/services/`：SQL 查询与统计逻辑
- `backend/models/`：数据模型
- `database/`：建表与导入脚本（后续实现时补充）
- `docs/`：报告素材、图表说明、答辩材料

## 4. 功能设计（10 个核心分析）

### 4.1 分析列表（每条都属于不同角度）
1. 年度犯罪总量趋势（时间-年度）
2. 一周七天分布（时间-星期）
3. 24 小时分布（时间-小时）
4. 犯罪类型占比（类别结构）
5. 行政区犯罪对比（空间-区域）
6. 逮捕率整体占比（执法结果）
7. 家庭暴力占比（案件性质）
8. 发生地点类型 Top10（场景结构）
9. 指定年份月度趋势（时间-月份）
10. 各犯罪类型逮捕率对比（类别 × 执法结果）

### 4.2 每条分析的统一展示规范
- 图表标题：问题导向（例：哪些小时段是高发时段？）
- 指标定义：统计口径明确（总数、比例、TopN、同比）
- 交互能力：悬浮提示、筛选器、时间范围切换、图例开关
- 结论区：至少 1 条结论 + 1 条解释 + 1 条可继续追问方向

## 5. 前后端交互方案

### 5.1 API 规范（规划）
- 路径风格：`/api/v1/...`
- 返回结构统一：
  - `code`: 业务状态码
  - `message`: 可读信息
  - `data`: 图表数据主体
  - `meta`: 参数回显、统计口径、缓存命中等
  - `request_id`: 链路追踪
- 参数规范统一：
  - 时间参数：`start_date`, `end_date`, `year`
  - 筛选参数：`crime_type`, `district`, `ward`
  - 排序/限制：`limit`, `order`

### 5.2 数据流
- 前端页面触发筛选条件
- `service` 层发起请求
- FastAPI 路由接收参数并校验
- 服务层执行聚合 SQL
- 返回图表数据
- D3 组件渲染并更新交互状态

### 5.3 错误处理策略
- 参数错误：返回明确字段与示例值
- 业务空数据：返回空数组并提示“当前条件无数据”
- 系统异常：记录日志并返回统一错误码
- 前端降级：图表区域显示空态/重试按钮，不让页面整体崩溃

## 6. D3 可视化实现策略

### 6.1 组件分层
- 通用容器：负责尺寸监听、加载态、空态、错误态
- 基础图表：Line/Bar/Pie 三类基组件
- 业务图表：基于基础图表封装业务字段映射

### 6.2 渲染策略
- 使用 React + `useRef` 驱动 D3 绘制
- 首次挂载创建 SVG，数据更新走 update 模式，避免整图重绘
- Tooltip 与高亮交互统一封装，保证视觉与交互一致

### 6.3 性能策略
- 默认限制 TopN，避免过长分类轴
- 视图切换懒加载
- 后端尽量完成聚合，前端只做映射与渲染

## 7. Electron 桌面化策略

### 7.1 安全基线（实现阶段必须遵守）
- `contextIsolation: true`
- `nodeIntegration: false`
- preload 仅暴露白名单 API
- 禁止不受控外链导航与新窗口

### 7.2 运行模式
- 开发模式：前端 dev server + 后端本地服务并行
- 演示模式：一键启动本地后端并加载桌面 UI
- 打包模式：生成 Windows 可执行安装包用于汇报现场

### 7.3 发布关注点
- 打包后静态资源路径
- 环境变量与 API 地址切换
- 汇报现场离线可运行能力

## 8. 数据库与 SQL 方案

### 8.1 表设计原则
- 保留原始字段，必要时新增解析字段（如小时、星期）
- 对高频筛选字段建立索引（日期、类型、区域、逮捕标记等）
- 数据导入后进行字段合法性校验与缺失值处理

### 8.2 SQL 编写原则
- 每个分析一个主查询，避免“换参数算同一分析角度”
- SQL 可复现：固定统计口径与时间边界
- 复杂查询优先拆成可读子查询，保障答辩可解释性

## 13. 主要风险与规避

- 风险：10 个分析角度被判定重复  
  规避：先冻结“角度定义”，每项都给出不同维度解释。

- 风险：联调晚导致图表展示不稳定  
  规避：先做接口契约与模拟数据并行开发，尽早联调。

- 风险：打包后资源路径/接口地址异常  
  规避：开发期就建立 dev/prod 配置分离与打包自检清单。

- 风险：答辩时只讲技术不讲洞察  
  规避：每张图绑定“结论话术”，按问题-证据-结论表达。
