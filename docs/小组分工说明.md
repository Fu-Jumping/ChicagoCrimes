## 一、项目当前完成情况（给大家看的总览）

- **数据库**：用的是 MySQL， crime 数据已经按课程要求落在关系型数据库里（有主表，还有为了查询更快而准备的汇总表，写报告时可以说「主表存明细、汇总表加速统计」即可，不必深究原理）。
- **后端**：本地有一个 **Python 写的接口服务**（FastAPI），跑起来后像一个小网站一样提供「要数字、要列表」的地址；桌面软件通过 **[http://127.0.0.1:8000](http://127.0.0.1:8000)** 这类地址去要数据。
- **桌面客户端**：是一个 **Electron 桌面程序**（可以当成「包了一层浏览器的专用软件」）。里面有总览、趋势、类型、区域、地图等页面，图表大多是 **D3** 画的，可以筛选年份、类型、区域等。
- **第一次使用**：如果电脑里还没配好库或没导好数据，软件里会有 **设置向导**，按步骤选 CSV、导入、建汇总等（这部分需要有人专门测一遍是否顺畅）。

**当前状态结论**：  
从老师作业的角度看，**「能存数据、能查能分析、能画图展示」这条主线已经打通了**，作品**已经具备可以交作业的形态**。  
现在还差的是：**书面作业里要交的文字和 SQL**、**数据字段说明**、**有人实测软件有没有坑**、**汇报用的 PPT**。这些是大家接下来要补的「小块」，以及一起确认 **客户端在你我的电脑上是否都能正常用**。

## 三、各岗位详细说明（按人）

### 仲禹潼（任务 1：数据初步分析 · 字段说明）

**目标**：让老师看到我们**真的看过数据长什么样**，每个字段大概是什么意思、在数据库里是什么类型。

**建议做法（保持简单）**：

打开

**路径**：`backend/sql/schema_crimes.sql`

里面从 `CREATE TABLE ... crimes` 开始，每一行就是一个字段，例如 `id BIGINT`、`date DATETIME`、`primary_type VARCHAR(100)` ……  
**字段名 + 类型**都写在这里了

对照用直观的语言和理解写一下：

- **列 1**：字段名（英文，和数据库里一致）  
- **列 2**：中文含义
- **列 3**：类型（可写「数字 / 文字 / 日期时间 / 是否」，可括号备注如 `INT`、`VARCHAR`）

**完成**：一张/页**清晰可读的字段说明表/文字**，能对照本项目实际使用的表，可以提供给PPT作为素材。

### 戴宇涵、马春暖（任务 2、3：SQL + 分析结论）

**目标**：满足作业里 **「用 SQL 分析 + 用文字写结论」**，且 **分析条数够**（老师要求不少于 10 条，且**不同分析角度**；同一种分析只换「犯罪类型」算一条，不要凑十条换类型）。

**建议分工**：

- 两人**先商量好谁写哪几条角度**，避免重复。例如一人侧重 **时间**（年趋势、月、星期、小时），另一人侧重 **类型 / 区域 / 逮捕 / 家暴 / 地点** 等。  
- 每条分析固定格式，**两行就够**：  
  1. **SQL**：正常语句
  2. **结论**：用 **一两句人话** 说结果（例如「某年比某年多」「周末比周几多」「前几种类型占大多数」，可以分析一两个，也可以分析好几组）

**完成**：  

- 两人合计 **至少 10 条（可15条左右）不同角度的「SQL + 结论」，**可以提供给PPT作为素材

### 林晨（封装客户端首启流程测试 + 问题记录）

**目标**：确认 **封装好的桌面客户端**（安装包）在 **新电脑 / 第一次使用** 时，从 **打开软件 → 设置向导 → 导入数据 → 进入主界面能看到图** 这条路**能走通**。

注：当前功能**基本已经做完**，**测试若一切正常，理论上交作业完全够用且够好，就保持现状，不必再动代码，如果发现明显问题可以尝试解决**。

**推荐顺序（先用户，后开发）**：

1. **第一步：只当普通用户，测封装版本**
  - 用**打包好的客户端**完整走一遍首次流程。  
  - 可对照 `docs/user-guide.md` 里 **「首次启动向导」**（不用研究，边做边看）。  
  - 关注：是否会正常出现向导、导入能否完成、完成后能否进主界面、各页面点一点是否报错或白屏。
2. **第二步：测出问题，再进开发环境**
  - 若封装版某步失败，再在本机用 **开发方式**（如 `python start.py dev` 与前端 `npm run dev`）复现，并查看 **仓库里的文档**（如 `docs/user-guide.md`）和**相关源码**，帮助定位原因。  
  - 把现象记清楚：**第几步、点了什么、报错原文或截图**，方便大家一起看。
3. **第三步：关于改不改代码**
  - **能不大改就不大改**；**能不改就不改**，当前的项目如果没问题，对于交这个作业已经完全够用了，尽量减少进一步迭代的工作量。  
  - 仅当出现**明确影响使用**的问题时，再考虑修改。

**过程记录（建议一张表）**：**步骤 | 预期 | 实际 | 是否正常 | 截图或报错原文**。

**完成**：  

- 一份 **首次启用测试**结果，可以提供给PPT作为素材；  
- **问题清单（如有）**；  
- 若有修改代码，可以进行记录。
